# 设计优化 · 第三轮：运维与体验闭环（Ops & Experience Closed-loop）

> 目标：补齐"便捷 / 高效 / 稳定"的最后一公里——让系统**自举、自证、自愈**，普通人也能零门槛拥有安全、可维护的部署。

## 1. 待补齐的体验断层
- PIN 锁屏依赖浏览器 `crypto.subtle`，**只在 HTTPS/localhost 安全上下文可用**；用户用 `http://IP:8080` 访问会卡死在"确认 PIN"。
- 首次使用要手改 `.env`、手配 caddy、手设 GHCR 包可见性——极客向。
- 出问题只能翻容器日志，没有面向用户的自检。
- 升级 = 重新部署，无平滑、无回滚点。

## 2. 第三轮方案

### 2.1 首次运行向导（First-run Wizard）
- 数据库为空时，前端自动进入向导页（不再要求先输 PIN）：
  1. 创建管理员账户（邮箱+密码）或设置本地 PIN（二选一；家庭多人走账户，单人走 PIN）。
  2. 向导内自动生成 `JWT_SECRET`/`ENGINE_API_TOKEN` 并写入挂载卷（用户全程无感）。
  3. 可选：是否开启"家庭成员邀请"。
- 彻底消灭"手改 .env"。

### 2.2 开箱即用的安全上下文（Auto-HTTPS）
- All-in-One 镜像内置 caddy，**自动获取证书**：
  - 检测到 Tailscale 时，调用 `tailscale cert` 自动签发 `*.ts.net` 证书 → `https://机器名.ts.net:8080` 直接是绿锁，PIN 立即可用。
  - 无 Tailscale 时，生成**本地受信自签证书**并提示用户信任一次（或提供"仅局域网/仅本机"模式，PIN 在 localhost 始终可用）。
- 提供环境变量 `DISABLE_HTTPS=1` 的逃生口（仅限纯内网且用户明确知晓风险）。

### 2.3 自检与可观测（Self-diagnostics）
- 内置 `/healthz` 页面（人类可读）：展示 core 进程、数据库连通、静态前端、实时通道、证书状态，绿/黄/红一目了然。
- 前端"设置 → 系统状态"内嵌同样面板，异常项给出**具体操作建议**（如"caddy 未获得证书，请检查 Tailscale"）。
- 结构化日志统一前缀 + 级别，便于抓关键行。

### 2.4 一键升级与回滚
- 镜像 tag 同时推 `:latest` 与 `:sha-<commit>`；compose 用 `:latest` 但记录当前 digest。
- 提供"升级前自动备份 `/data`"开关（默认开），升级失败可一键回滚到上一 digest。
- PWA 离线缓存，弱网/重启间隙仍可查看已加载数据。

## 3. 本轮收益与遗留
**已解决**：零门槛安全访问、零手工配置、可读自检、平滑升级。
**遗留**：本轮偏"体验外壳"，若架构仍混乱（双后端），自检面板要监控的组件仍多、向导要兼容两种模式逻辑，反而增加复杂度。→ 故第三轮必须**建立在第二轮的内核统一之上**才最省。

## 4. 判定
- 精简：★★★（本身不加实体，但依赖第二轮才不反增复杂度）
- 稳定：★★★★（自检+备份+回滚）
- 高效：★★★★（向导省下大量人工排错）
- 便捷：★★★★★（普通人可用的安全部署）
