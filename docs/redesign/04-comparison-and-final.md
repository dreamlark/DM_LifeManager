# 三轮方案对比分析 & 最终设计方案（整合）

## 一、四维对比矩阵

| 维度 | 第一轮（部署收敛+UI表层） | 第二轮（架构内聚） | 第三轮（运维体验闭环） |
|---|---|---|---|
| **精简** | ★★☆ 部署层精简，代码层未动 | ★★★★ 1进程/1库/1客户端 | ★★★ 依赖第二轮才不反增 |
| **稳定** | ★★★ 少服务=少故障点 | ★★★★ 故障域减半、单写路径 | ★★★★ 自检+备份+回滚 |
| **高效** | ★★☆ 启动快，内部冗余在 | ★★★★ 少进程间转发/双写 | ★★★★ 向导省人工排错 |
| **便捷** | ★★★★ NAS一步部署 | ★★★ 部署已简，运维待补 | ★★★★★ 零门槛安全部署 |
| **改动量/风险** | 低（包一层+换皮） | 高（合后端+迁数据） | 中（向导+证书+自检） |
| **可独立交付** | ✅ 立即可做 | ⚠️ 建议先有第一轮容器 | ❌ 须建在第二轮之上 |

## 二、关键洞察
1. **三轮是递进依赖的**：第一轮解决"看得见"的摩擦；第二轮消除"摸得着"的复杂度根因；第三轮把前两轮的能力包装成"普通人可用"。单独做第三轮会在混乱架构上叠体验，反而更乱。
2. **最大杠杆点是第二轮的内核统一**：它同时改善精简/稳定/高效三项，且让第一轮和第三轮都更轻。
3. **第一轮性价比最高、风险最低**，应**最先落地**作为稳定基线；第二轮/第三轮在其之上增量推进，每步可回滚。

## 三、最终设计方案（整合三轮，遵循"精简·稳定·高效·便捷"）

> 一句话：**一个容器、一个内核、一个前端、零必填配置、开箱 HTTPS、首次向导。**

### 3.1 部署拓扑（整合第一轮 + 第三轮）
- **1 个镜像 `ghcr.io/dreamlark/dm-life:latest`**，内含：
  - `packages/core`（单进程 tRPC + 单库 PGlite + 统一实时）← 第二轮
  - 静态前端（构建自 `packages/web-collab`）← 第一轮
  - 内置 caddy（自动证书）← 第三轮
- **1 个 compose `docker-compose.simple.yml`**：1 服务、1 端口（8080，HTTPS 可选 443）、1 卷 `/data`、**零必填 env**。
- 缺失密钥自动生成并持久化；GHCR 包默认 Public（或提供 `read:packages` 登录说明）。

### 3.2 内核（整合第二轮）
- 删除 `packages/engine` 与 `packages/server` 进程边界 → 合并为 **`packages/core`**：
  - 单 `initTRPC`、单 PGlite 库、单事件总线、单实时通道（WS，SSE 降级）。
  - 个人模式 = 自动创建的本地单成员家庭；协作模式 = 同一模型的多成员家庭。
- 前端单一 `trpc` 客户端，mode 仅控制"能力开关"。

### 3.3 前端（整合第一轮 + 第二轮）
- 弃用 `packages/web`；`packages/web-collab` 为唯一前端。
- 单一主题令牌集 + 单一 `AppShell`；所有业务页只写一份。
- PIN/账户在首次向导中配置；UI 在所有模式下视觉一致。

### 3.4 体验闭环（整合第三轮）
- 首次运行向导（建账户/PIN、自动生成密钥）。
- 自动 HTTPS（Tailscale 证书优先，自签兜底，localhost 始终可用）。
- `/healthz` 自检面板 + 设置内系统状态。
- 升级前自动备份 `/data`，支持回滚到上一 digest；PWA 离线。

## 四、分阶段交付路线（自主决策版）

| 阶段 | 内容 | 风险 | 本会话交付 |
|---|---|---|---|
| **P1（基线）** | All-in-One 单容器 + 零必填配置；弃用 `web`；统一主题 + 单 AppShell | 低 | ✅ 实现 |
| **P2（内核）** | engine/server 合并为 core（单库 PGlite、单客户端、单实时） | 高 | 📄 设计 + 增量计划（分阶段、可回滚） |
| **P3（体验）** | 首次向导 + 自动 HTTPS + 自检 + 升级回滚 | 中 | 📄 设计（建于 P2 之上） |

> **本会话先落地 P1**（最高性价比、立即可用、为 P2/P3 打基础），P2/P3 以完整设计文档交付并给出可执行的增量实施步骤。这样既不破坏当前可运行系统，又把"精简/稳定/高效/便捷"的方向和落地路径全部锁定。

## 五、收益预期（最终态 vs 现状）
- 部署文件：4 服务 + 3 镜像 + `.env` + `caddy.conf` → **1 段 compose**。
- 运行实体：engine + server + frontend + caddy → **core + caddy**（进程数 4→2）。
- 数据库：sql.js + PGlite → **PGlite 单库**。
- 前端包：web + web-collab → **web-collab 单一**。
- 配置必填项：JWT_SECRET + ENGINE_API_TOKEN → **0**。
- 安全访问门槛：手动证书 → **自动 HTTPS（Tailscale/自签）**。
