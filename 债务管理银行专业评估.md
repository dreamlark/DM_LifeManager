# 债务管理模块 · 银行专业视角评估报告

> 评估对象：`dm-life` 财务模块债务管理（引擎 `schedule.ts` / `repository.ts` / `command.ts`，前端 `FinancePage.tsx`）
> 评估维度：债务分类、利率管理、还款进度追踪
> 评估立场：商业银行个金 / 信贷风控视角，兼顾个人财务管理（PFM）可用性

---

## 0. 结论速览

| 维度 | 当前成熟度 | 核心缺口 | 优先级 |
|------|-----------|---------|--------|
| 债务分类 | ★★★☆☆ | 担保方式/币种/合同号/优先级缺失，房贷与抵押贷概念重叠 | 中 |
| 利率管理 | ★★★★☆ | 重定价机制未自动化（LPR 品种/周期/对日），罚息/实际年化(IRR)缺失 | 高 |
| 还款进度追踪 | ★★★★☆ | 「计划进度」与「实际已还」脱节，缺优化建议与利息节省测算 | 高 |

**总体评价**：底层计算引擎（`getRepaymentSchedule` 支持 4 种还款方式 + 分段利率 + 提前还款）已具备银行级骨架，架构（事件溯源双写 + SSE 实时刷新）扎实；主要短板在**业务语义的完整度**与**计划/实际的勾稽**，而非算法正确性。本次新增的「双层本金偿还进度」已修正此前时间进度低估本金偿还的问题。

## 1. 债务分类（Debt Classification）

### 1.1 现状
- `debtType` 枚举：`mortgage`(房贷) / `credit_card`(信用卡) / `consumer`(消费贷) / `secured`(抵押贷) / `other`(其他)
- `repaymentMethod`：`equal_installment` / `equal_principal` / `interest_first` / `fixed_rate`
- `status`：`active` / `paid`（本次重构新增 `done` 用于一次性/已结清钟）
- 债权人/机构存于 `creditor` 字段

### 1.2 银行视角问题与缺失
1. **担保方式与产品类型混淆**。在银行口径中，「房贷」本质就是「抵押贷（不动产抵押）」，二者是**包含关系**而非并列。当前 `mortgage` 与 `secured` 并列，既重叠又无法表达「信用消费贷 vs 抵押消费贷」这类细分。建议将**担保方式**（信用/抵押/质押/保证）抽离为独立维度，与产品类型正交。
2. **缺币种（currency）**。全部隐含人民币。若系统目标含外币留学贷/美元债，币种缺失会导致利率口径（LPR 仅人民币）、汇总金额错乱。
3. **缺合同号 / 账户标识**。银行对账、提前还款预约、征信核对均依赖唯一合同号；当前仅有 `creditor` 文本，无法精确 reconciliation。
4. **缺债务优先级 / 风险标签**。个人债务优化（雪崩法/滚雪球法）需要先区分利率高低与心理阈值；系统未提供 `priority` 或 `strategy_tag`。
5. **信用卡建模不足**。信用卡是「循环授信 + 最低还款 + 免息期」的特殊形态，用 `termMonths` + 等额本息近似会严重失真（实际是 daily compounding + 最低还款额）。当前 `credit_card` 仅是一个 label，未特殊计算。

### 1.3 优化建议
- **维度正交化**：`product_type`（房贷/信用卡/消费贷/经营贷…）× `collateral_type`（信用/抵押/质押/保证）× `currency`（CNY/USD…）。
- 增加 `contract_no`、`priority`（int）、`is_revolving`（boolean，信用卡/循环贷）。
- 信用卡单独建模：引入 `credit_limit`、`statement_day`、`due_day`、`min_payment_ratio`、`grace_days`，按日计息而非按月等额。
- 分类支持「标签体系」而非单一枚举，便于后续按监管口径（自营/助贷/联合贷）统计。

## 2. 利率管理（Interest-Rate Management）

### 2.1 现状（已较完整）
- `rateType`：`fixed`(固定) / `benchmark`(基准利率) / `lpr`(LPR)
- `fixed` → 用 `apr`；`lpr`/`benchmark` → 有效年利率 = `baseRate` + `rateSpread`（引擎 `getEffectiveAnnualRate`）
- 支持 `rateAdjustments[]`（生效日 + 新利率，分段利率）与 `rateAdjustDay`（重定价日）
- 引擎能正确在利率变动月重算月供（`rateChanged` 触发重算）
- 提前还款 `prepayments[]` 支持 `reduce_term` / `reduce_payment`

### 2.2 银行视角问题与缺失
1. **重定价机制未自动化（最关键缺口）**。真实 LPR 浮动贷：在**重定价日**取「该日公布的同期限 LPR + 加点（永久锁定）」重定价，之后一年内利率不变。当前 `rateAdjustments` 是**用户手动录入的 (生效日, 新利率) 列表**——需要用户自己盯着央行每月 20 日公布的 LPR 去逐期填，既不现实也易错。缺少：
   - 挂钩的 **LPR 期限品种**（1 年期 / 5 年期以上）
   - **重定价周期**（年 / 半年 / 季）
   - **重定价日类型**（对年对月对日 / 固定日历日，如每年 1 月 1 日）
   - 可选「自动取数」对接 LPR 发布源
2. **加点（spread）口径正确但无审计**。spread 在签约时锁定、仅 base 浮动——这点引擎逻辑对，但 UI 未在重定价时直观展示「本次重定价：LPR 3.95% + 加点 0.50% = 4.45%」的来龙去脉。
3. **罚息 / 逾期利率缺失**。信贷资产必然有逾期罚息（通常上浮 30%–50%），系统无法模拟违约成本，削弱了「按时还款」的激励可视化。
4. **名义利率 vs 实际年化（IRR/EAR）缺失**。等额本息下，银行披露的 **APR（内部收益率）** 显著高于名义年利率（资金被按月占用）。当前仅存名义 `apr`，未计算 IRR，不利于横向比较不同还款方式的真实成本。
5. **利率历史可视化弱**。虽然 `rateAdjustments` 已是结构化数据，但前端未在债务详情展示「利率变动曲线」。

### 2.3 优化建议
- 引入**重定价规则对象**：`repricing = { benchmark: 'LPR_1Y'|'LPR_5Y'|'PBOC_BASE', spread, cycle: 'Y'|'M', anchor: 'anniversary'|'fixed_date', fixedDate? }`，引擎按规则自动生成 `rateAdjustments`，用户无需手填。
- 增加 `penalty_rate`（罚息上浮比例）与逾期状态联动。
- 计算并展示 **IRR/实际年化利率**（对等额本息用现金流 IRR；对先息后本近似 EAR），作为「真实成本」指标。
- 债务详情页增加**利率变动时间轴**与「当前执行利率」徽标（引擎已算 `currentRate`，前端未充分暴露）。

## 3. 还款进度追踪（Repayment Progress Tracking）

### 3.1 现状（本次已增强）
- 单笔与整体**双层进度**（`debtProgressSummary`）：整体按本金加权，单笔展示 `principalProgress`（已还本金/原始本金）+ 期数进度
- `DebtScheduleView` 展示**已还清单**（绿）/ **待还清单**（琥珀），数据来自 `debt_payment` 流水
- 引擎 `autoGenerateMonthly` 会按扣款日自动生成还款流水

### 3.2 银行视角问题与缺失
1. **计划进度 vs 实际已还「两张皮」（最严重）**。当前进度条来自 `schedule`（**计划值**，由起贷日推算），而「已还/待还清单」来自 `debt_payment` 交易（**实际值**，依赖 `autoGenerateMonthly` 跑过且用户未漏记）。二者未勾稽：若自动生成未运行或某月漏还，清单为空但进度条仍显示 60%——用户看到「还了 60%」却点开清单什么都没有。应统一为「实际已还本金 = Σ实际还款交易本金」，计划值仅作对照。
2. **提前还款的利息节省未量化**。引擎在 `prepayments` 下会重算月供/期限，但未输出「因提前还款节省的利息总额」「缩短的期数」，而这是用户做提前还款决策最关键的信息。
3. **缺剩余利息总额预测**。进度条只显示本金维度，未展示「剩余待付利息」，而利息才是债务的真实负担。
4. **缺债务优化建议引擎**。银行 PFM 常给「雪崩法（先还高利率）/ 滚雪球法（先还小额）」建议。当前系统只是陈列，不主动优化。
5. **月供变动不显性**。利率重定价后月供会变，`schedule` 已含但 UI 未突出「第 N 期起月供由 X 变为 Y」。
6. **精度与边界**：`principalProgress` 用 `principal - remaining` 推导，已做 `< 1` 容差；但 `remainingPrincipal` 为 0 时进度应为 100%（已处理）。一次性（无期限）债务已被 `debtProgressSummary` 过滤，合理。

### 3.3 优化建议
- **勾稽实际**：新增 `actualPaidPrincipal`（Σ `debt_payment` 本金），进度条以实际值为主、计划值为副，二者差异用提示色标出。
- **提前还款收益卡**：输出 `interest_saved`、`term_shortened`、`new_monthly`。
- **利息维度**：进度区补充「剩余利息」与「累计已付利息」。
- **优化建议**：按 IRR/余额生成「优先偿还序列」建议（雪崩/滚雪球可切换）。
- **月供时间轴**：在清单中标注月供变动节点。

## 4. 优先级路线图（建议）

| 优先级 | 项 | 收益 | 工作量 |
|--------|----|------|--------|
| P0 | 重定价规则自动化（LPR 品种/周期/对日）+ 自动生成 rateAdjustments | 消除最大手工错误源 | 中 |
| P0 | 计划进度 ↔ 实际已还勾稽（actualPaidPrincipal） | 解决数据可信度根本问题 | 中 |
| P1 | 实际年化利率 IRR/EAR 披露 | 真实成本可比 | 小 |
| P1 | 提前还款利息节省 + 期限缩短量化 | 决策支持 | 小 |
| P1 | 债务优先级 + 雪崩/滚雪球建议 | PFM 差异化价值 | 中 |
| P2 | 担保方式/币种/合同号维度正交化 | 风控合规 | 中 |
| P2 | 信用卡循环贷独立建模 | 建模正确性 | 大 |
| P2 | 罚息/逾期利率 + 利率变动时间轴 | 完整度 | 小 |
